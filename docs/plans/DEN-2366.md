# DEN-2366 — Plan file convention for bear-metal PRs

## Goal

Teach the bear-metal worker to:

1. When opening a new PR, commit a task plan to `docs/plans/DEN-<TICKET ID>.md` as
   part of the same PR so reviewers (and baloo) can compare the diff against an
   explicit, written plan.
2. When iterating on PR feedback, treat baloo's fidelity report as a first-class
   review thread: read each gap, judge whether it is meaningful (real divergence
   from the ticket spec or the committed plan) or noise, and either fix the
   meaningful gaps or reply explaining why the gap is not actionable.

## Files touched

- `bear-metal/src/worker/prompts.ts` — extend `buildWorkerPrompt` for both
  `state === "new"` and `state === "iteration"` to add the plan-file step and
  the baloo fidelity-report handling step. The plan path is derived from
  `context.ticketId` (`docs/plans/${ticketId}.md`).
- `bear-metal/src/worker/worker-contract.test.ts` — assert that the new state
  prompt mentions `docs/plans/DEN-2.md` and "task plan", and that the
  iteration prompt mentions `baloo`, `fidelity report`, and
  `docs/plans/DEN-1.md`.
- `bear-metal/docs/plans/DEN-2366.md` — this file, the plan itself, shipped in
  the PR per the new convention.

## Non-goals

- No changes to baloo's review format, no changes to how PR bodies are
  generated, no changes to the tool contract (`wrote_code`,
  `respond_to_ticket_reporter`, etc.). The plan file is committed by the worker
  via normal `wrote_code` flow; nothing reads or parses it inside bear-metal.

## Verification

- Updated `worker-contract.test.ts` covers both prompt branches.
- Local sanity check with a tsx one-liner confirms the rendered prompts contain
  the expected substrings for both `new` and `iteration` states.
