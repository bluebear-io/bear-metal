# DEN-2518: Fix "Review-thread tools are only supported for single-PR iterations"

## Problem

`requireSinglePr` in `bear-metal/src/worker/pi.ts` threw for any iteration with more than one open PR, so all review-thread tools (`agree_with_github_message`, `disagree_with_github_message`, `mark_github_message_completed`, `respond_to_comment_writer`) failed on cross-repo tasks (e.g. PRs in both `blueden` and `handler`).

The guard existed because the call sites had no way to know which PR a given thread/comment id belonged to.

## Approach

GitHub review-thread and issue-comment node IDs are globally unique GraphQL node IDs — each id belongs to exactly one PR. So we can resolve `id → PR` with a precomputed map and drop the single-PR guard entirely.

## Changes

- `bear-metal/src/worker/pi.ts`
  - Build `commentToPr: Map<string, PullRequestRef>` in the same loop that builds `commentMap`, populated from both `unresolvedReviewThreads[].id` and `issueComments[].id` for every PR in `input.context.pullRequests`.
  - Replace all 4 `requireSinglePr(input.context.prs)` call sites with `commentToPr.get(params.id)`, throwing `Unknown comment id: <id>` when missing (same fail-fast contract as `commentMap`).
  - Delete `requireSinglePr` (no remaining callers).
  - Remove the stale "enforced by `requireSinglePr`" comment on `unresolvedThreadsFor`; the existing `(owner, repo, number)` lookup already handles multi-PR.
- `bear-metal/src/worker/pi.test.ts`
  - Update existing tests that referenced thread ids not present in the PR context (they only passed because `requireSinglePr` ignored the id).
  - Add: `routes review-thread tools to the correct PR when multiple PRs are in context` — two PRs, verifies each tool call hits the right PR.
  - Add: `throws when review-thread tool is called with an unknown comment id` — regression guard for the new lookup.

## Verification

- `npm test` — all tests pass (141, +2 new).
- `tsc --noEmit` clean.
