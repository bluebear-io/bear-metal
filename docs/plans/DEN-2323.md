# DEN-2323 — Backfill tool: rebuild dashboard DB from Linear + GitHub

> Linear: https://linear.app/bluebearsecurity/issue/DEN-2323

Build a one-shot CLI under `src/backend/backfill/` that walks every Linear ticket delegated to the Bear Metal agent, enriches it with GitHub PR + CI data, and writes a coherent ticket-bundle to the dashboard SQLite DB. Skip-on-conflict per ticket so re-runs are resumable.

## Constraints (from the ticket)

- **No changes to `src/manager/` or `src/worker/`.**
- **Additive only** to `src/shared/integrations/`: new methods, new optional fields on returned types; no breaking changes to existing signatures or shapes.
- **Skip on conflict** at the `tickets` row — leave all dependent rows alone for tickets that already exist.
- **Stable, deterministic synthetic IDs** so dependent rows also collide cleanly on re-run.
- **Reuse** existing integration clients; do not write new Linear/GitHub HTTP code.

## Data flow

```
Linear (delegated tickets, all states)
  → enrich each ticket with GitHub PRs (lookup by Linear's gitBranchName across the App's installation repos)
    → enrich each PR with the latest check runs
      → map to row bundle { ticket, runs[], pull_requests[], ci_runs[], events[] }
        → write each bundle in a single transaction with skip-on-existing-ticket
```

Plus: one upserted `workers` row (`wk_backfill`) referenced by every synthesized run.

### T7 — Writer: transactional insert with skip-on-conflict

**Files:** `src/backend/backfill/writer.ts`, `src/backend/backfill/writer.test.ts`.

- `ensureBackfillWorker(db): void` — upsert the `wk_backfill` worker row (allowed exception to skip-on-conflict per the ticket).
- `writeBundle(db, bundle): { written: boolean }` — inside a transaction:
  1. Check if `tickets.id` already exists; if yes, return `{ written: false }`.
  2. Insert in FK-safe order: `tickets` → `runs` → `pull_requests` → `ci_runs` → `events`.
  3. Return `{ written: true }`.
- Unit tests against an in-memory SQLite (mirror the seeder test pattern): fresh DB writes everything; second call with same bundle is a no-op; new ticket with existing-ticket present writes only the new one.

**Commit:** `feat(backend): backfill — DB writer (DEN-2323)`
