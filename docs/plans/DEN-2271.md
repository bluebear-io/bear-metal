# DEN-2271: bear-metal observability dashboard

## Overview

A read-only dashboard to observe the bear-metal background coding agent: which Linear
tickets it picked up, how each attempt (run) progressed across the manager → worker → CI
loop, the PRs it opened, and worker health. The MVP is driven by **mock data seeded into
SQLite**; the manager will later write real data conforming to the same schema.

**Linear:** [DEN-2271](https://linear.app/bluebearsecurity/issue/DEN-2271/design-ui-application) (parent: [DEN-2266](https://linear.app/bluebearsecurity/issue/DEN-2266/bear-metal-epic))

## Context

### Bear Metal workflow (the system we observe)
- **Manager** polls Linear (label `bear-metal`) for candidate tickets. On finding one it
  reviews it, builds context, and dispatches it to an available **worker** (pool of N).
- A **worker** starts fresh, gathers context from the Linear ticket + GitHub. For a new
  ticket it creates a branch, works the task, updates Linear + GitHub, opens a PR, and
  reports back to the manager.
- On PR push, **CI** runs the tests. Pass → ticket completed. Fail → Linear ticket is
  updated and delegated back to Bear Metal.
- The manager re-dispatches an open ticket (with context incl. the existing branch) to
  whatever worker is free. The worker resolves the open items, updates Linear + GitHub,
  pushes, reports back.
- The manager enforces a **per-ticket iteration limit** (e.g. ≤5 attempts).
- A worker may die/hang; the manager **monitors workers** and stops any worker running
  longer than M minutes, marking it accordingly.

So a ticket has **many attempts (runs)**, each handled by a worker, each possibly
followed by a CI result that drives the next attempt.

### Current repo state (origin/main)
- TypeScript, single npm package, ESM (NodeNext), `rootDir: src`, build via `tsc`,
  tests via **vitest**. HTTP via **Express**, logging via **pino**.
- Components are folders under `src/` with a barrel `index.ts`: `src/shared`, `src/worker`,
  `src/manager`. Real worker/LLM logic and the ticket state machine are not implemented yet.
- Concrete domain types already exist and the schema below mirrors them:
  - Linear `Ticket` (`src/shared/integrations/linear/types.ts`): `id`, `identifier`,
    `title`, `description?`, `url`, `branchName`, `status {name, type}`, `labels[]`.
  - GitHub `PullRequest` (`src/shared/integrations/github/types.ts`): `number`, `title`,
    `headRef`, `state` (`open|closed`), `draft`, `merged`, `url`.

### The console design system (reuse target)
Not a shared package — Tailwind CSS 4 with BlueBear CSS-variable tokens (light/dark) in
`console/src/app/globals.css`, `lucide-react` icons, `recharts`, custom React 19 components.
Reuse = port the tokens + conventions, not import a library.

## Decisions (approved)

| Topic | Decision |
|-------|----------|
| Purpose | Read-only observability dashboard. The dashboard never writes agent state. |
| Data ownership | We define the schema as the contract. MVP seeds **mock data**; manager writes for real later. No shared packages with manager/worker yet. |
| Backend framework | **Express** (matches the repo's existing HTTP stack). |
| Structure | Both new components under `src/`: `src/backend/` (joins root package) and `src/ui/` (self-contained Vite app). |
| Frontend | React 19 + Vite + Tailwind 4 with ported BlueBear tokens, `lucide-react`, `react-router`, TanStack Query. |
| DB | SQLite via **Drizzle**, opened **read-only** by the backend. |
| Runtime / auth | Local-only, no auth for MVP; backend leaves a no-op auth middleware seam. |
| Data freshness | Manual refresh (load on open + refresh button). No polling/streaming. |
| MVP scope | Tickets list + ticket detail + a lightweight Workers panel. |

## Data Model (SQLite / Drizzle) — finalized

Field names mirror the existing Linear/GitHub types where they overlap. Timestamps are
epoch-ms integers. `*_json` columns hold serialized JSON. The backend opens the DB
**read-only**; the mock seeder uses a separate writable connection.

**`tickets`** — a Linear ticket tracked by Bear Metal
`id` (pk, Linear ticket id), `identifier` (`DEN-2271`), `title`, `description?`, `url`,
`branch_name`, `linear_status_name`, `linear_status_type`, `labels_json`,
`bm_status` (`discovered|dispatched|in_progress|pr_open|ci_running|ci_failed|completed|abandoned`),
`attempt_count`, `max_attempts`, `created_at`, `updated_at`, `completed_at?`

**`workers`** — the worker pool the manager dispatches to and monitors
`id` (pk), `name`, `status` (`idle|busy|stopped|dead`), `current_run_id?`,
`last_heartbeat_at?`, `started_at`, `updated_at`

**`runs`** — one dispatch/iteration of a ticket to a worker (ticket → many runs)
`id` (pk), `ticket_id → tickets`, `attempt_number`, `worker_id → workers?`,
`trigger` (`new|ci_failure|delegated_back`),
`status` (`dispatched|running|succeeded|failed|timed_out|crashed`),
`context_json?` (manager-provided context: branch, prior notes), `started_at?`,
`ended_at?`, `stop_reason?` (`completed|timeout|crash|error`), `error?`, `created_at`

**`pull_requests`** — mirrors GitHub `PullRequest`; one per ticket/branch, updated across attempts
`id` (pk), `ticket_id → tickets`, `number`, `title`, `head_ref`, `state` (`open|closed`),
`draft` (bool), `merged` (bool), `url`, `last_run_id → runs?`, `created_at`, `updated_at`

**`ci_runs`** — CI result for a run's PR push (drives complete vs delegate-back)
`id` (pk), `ticket_id → tickets`, `run_id → runs`, `pr_id → pull_requests?`,
`status` (`running|passed|failed`), `url?`, `summary?` (failing checks), `created_at`,
`completed_at?`

**`events`** — unified timeline across manager + worker + CI
`id` (pk), `ticket_id?`, `run_id?`, `worker_id?`, `source` (`manager|worker|ci`),
`type` (`ticket_discovered|dispatched|branch_created|progress|pr_opened|ci_started|ci_passed|ci_failed|delegated_back|worker_timeout|worker_crashed|ticket_completed|ticket_abandoned`),
`summary`, `payload_json?`, `created_at`

The model captures: iteration limits (`attempt_count`/`max_attempts`, `bm_status=abandoned`),
the CI → delegate-back loop (`ci_runs` + `runs.trigger`), and worker death/timeout
(`workers.status`, `last_heartbeat_at`, `runs.stop_reason`/`status`).

## API (Express)

- `GET /api/health`
- `GET /api/tickets` — list with latest run status, attempt_count/max, latest PR + CI; filterable by `bm_status`; newest first.
- `GET /api/tickets/:id` — ticket + all runs (with worker), PRs, CI runs, and merged event timeline.
- `GET /api/workers` — worker list with status, current ticket/run, runtime, timeout/dead flags.
- No-op auth middleware seam so real auth can be added later without rework.
- **Fail-fast** (repo rule): missing DB file or invalid rows → explicit error response and
  logged error; never a silent fallback or fabricated value.

## Frontend screens

- **`/` Tickets list** — table: identifier (Linear link), title, `bm_status` badge,
  attempts `N/max`, branch, latest CI result, PR link. Refresh button.
- **`/tickets/:id` Ticket detail** — header (Linear link, repo/branch, status, attempts),
  per-attempt runs with worker + outcome, PR(s), CI history, and the merged event timeline.
- **`/workers` Workers panel** — list of workers: status, current ticket/run, runtime,
  timeout/dead indicators.
- Port a **minimal** set of console-style primitives (PageHeader, status badge/`Label`,
  table, ThemeToggle) using the ported BlueBear tokens — not the whole console library.

## Risks

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| Schema drifts from manager's eventual writes | Medium | High | Mirror existing `src/shared` types; keep schema in repo as the shared contract; revisit when manager wires the DB. |
| `src/ui` browser code breaks the root `tsc` build | Medium | Medium | Exclude `src/ui` in root tsconfig; UI has its own toolchain. |
| Concurrent SQLite access (manager writing while UI reads) | Medium | Medium | Open DB read-only; SQLite WAL; surface read errors fail-fast. |
| Design-token drift from console | Low | Low | Port tokens verbatim from `globals.css`; document source. |
