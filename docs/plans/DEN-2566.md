# DEN-2566 ‚Äî Bear Metal: include all PRs in Slack notifications for multi-PR tickets

## Problem
`tryTransitionToWaitingForHuman` Slack path in `bear-metal/src/manager/scheduler.ts` only notifies the first PR (`decision.context.prs[0]!`). For cross-repo tickets, reviewers miss the other PRs.

## Approach
1. Generalize the Slack notification API to carry a list of PRs.
   - `PullRequestNotification` becomes `{ kind, prs: Array<{ pr: PullRequestRef; url: string }>, title, ticketId, ticketUrl, recipientEmail? }`.
   - `formatNotificationText`:
     - 1 PR ‚Üí keep existing wording (`:bear: PR opened <link> for ticket <ticket> ‚Äî title` and `Updated PR <link> for ticket <ticket> ‚Äî title`).
     - >1 PRs ‚Üí single message listing every PR link:
       - opened: `:bear: PRs opened for ticket <ticket|ID> ‚Äî Title\n<link1>, <link2>`
       - updated: `PRs updated for ticket <ticket|ID> ‚Äî Title\n<link1>, <link2>`
     - Preserve mrkdwn escaping for title / ticket id / owner / repo.
2. Scheduler `tryTransitionToWaitingForHuman` block in `scheduler.ts`:
   - For every PR in `decision.context.prs`, fetch its `getPrNotifiedAt` and live PR status (URL).
   - Split into `opened` (never notified) and `updated` (already notified) groups.
   - Send one Slack message per non-empty group (so a mixed state = 2 messages).
   - Mark each included PR as notified and emit a `user_notified` event per PR.
3. `WorkerSlack` interface uses the same `PullRequestNotification` shape ‚Äî update type.
