# DEN-2576 — bear metal healthcheck

## Problem

The bear-metal container healthcheck (`GET /api/health`) was unreachable during
startup whenever a worker environment builder was configured. The builder can
take up to 30 minutes (`TIMEOUT_MS = 30 * 60 * 1000`), and the previous startup
order called `app.listen` only after `runWorkerEnvironmentBuilder` and
`db.initSchema` finished. Docker/k8s healthchecks therefore failed for the entire
bootstrap window.

## Change

`runWorkerEnvironmentBuilder` is only required for the worker loop to run; the
HTTP API and UI do not depend on it. So the fix is just to reorder startup:

- `src/manager/index.ts`: initialize the DB and start the full Express app
  (`createApp` + `app.listen`) *before* `runWorkerEnvironmentBuilder`. The
  scheduler and task worker loops still start only after the builder completes.
  Test-ticket mode skips the server (preserves previous behavior); shutdown
  tolerates the no-server case.
- `src/manager/routes.ts`: removed the duplicate `/health` route from
  `createRouter` (the `/api/health` route mounted in `createApp` is the single
  source of truth).
- `src/manager/app.ts`: unchanged shape — still a single `createApp(db, maxIterations)`.

## Files

- `bear-metal/src/manager/index.ts`
- `bear-metal/src/manager/routes.ts`
- `bear-metal/src/manager/app.ts`

## Verification

- `npm run typecheck` — clean.
- Manual: `curl localhost:<port>/api/health` returns 200 immediately after the
  server starts, while `runWorkerEnvironmentBuilder` is still running.
