# DEN-2321 — Code review & cleanup: UI + backend

Scope: `src/backend/` and `src/ui/` only. Refactor only — no behavior changes.
Public API responses and UI rendered output stay identical. Tests stay green.

Changes are ordered low-risk → higher-risk. Each commit runs `npm test` and
`npm run typecheck` (and `npm run typecheck` inside `src/ui/`).

## 3. UI consistency (low risk)

5. **`src/ui/src/pages/TicketsListPage.tsx`** — the `useMemo<TicketListItem[]>`
   around `q.data ?? []` only stabilizes the empty-array identity. Inline it
   and pass `q.data ?? []` directly into the downstream `useMemo`s that already
   compute `counts` and `visibleTickets`. The `useMemo`s downstream depend on
   `tickets`, so we keep `tickets` as a `const` referenced once. No behavior
   change. Category: dedup / consistency.
6. **`src/ui/src/pages/TicketDetailPage.tsx`** — `QueryBoundary` already accepts
   `Error | null`. The `query.error instanceof Error ? query.error : null` guard
   was redundant; pass `query.error` through. (TanStack Query's `error` is typed
   as `Error | null` in v5.) Same for `ModelsPage`. Category: consistency.
7. **`src/ui/src/pages/TicketDetailPage.tsx`** — replace the stale `t.replaceAll`
   inline string transforms with `humanizeStatus`-style call via `format.ts`?
   Skipped (would expand format.ts surface area, three-similar-lines rule).

## 4. Repository perf — batched lookups (medium risk)

8. **`src/backend/db/repository.ts` — `listTickets`** — replace the per-ticket
   sub-queries (`runs`, `pull_requests`, `ci_runs`) with three batched queries
   that fetch all rows for the filtered ticket set at once, then group them in
   JS. Latest run is the row with the highest `(attemptNumber, createdAt)`
   tuple per `ticket_id`; latest PR is the row with the highest `updatedAt`
   per `ticket_id`; latest CI status is the row with the highest `createdAt`
   per `ticket_id`. Order of the response array stays identical (DB-side
   `ORDER BY tickets.createdAt DESC`). Category: perf.
9. **`src/backend/db/repository.ts` — `getTicketDetail`** — the per-PR review-
   thread query and per-CI-run checks query are N+1 in the number of PRs/CI
   runs for the ticket. Replace with one bulk query each, then group in JS.
   Workers map already loads all workers — leave the comment + a one-line note
   that this is OK for the small operator-facing DB. Category: perf.
10. **`src/backend/db/repository.ts` — `listWorkers`** — replace the per-worker
    `runs` + `tickets` round-trips with two batched lookups keyed by the union
    of `currentRunId` values. Same response shape. Category: perf.
