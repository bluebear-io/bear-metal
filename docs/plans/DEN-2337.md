# DEN-2337 — UI: productivity summary screen

> Linear: https://linear.app/bluebearsecurity/issue/DEN-2337

A single `/summary` page with **Last 24h / Last week / Custom** chips and six cards: Throughput, Pipeline health, Cost & LLM usage, Time, Top failures, What shipped. Every headline metric carries a period-over-period delta vs the prior same-length window.

## Decisions baked in

- **No schema changes** — every metric comes from existing tables (`tickets`, `runs`, `pull_requests`, `ci_runs`, `ci_checks`, `events`).
- **One server round-trip per page load.** `GET /api/summary?from&to` returns the whole `PeriodSummary` blob. Fan-out happens inside `Repository.getPeriodSummary`.
- **URL-driven period state.** `?from=<iso>&to=<iso>` on `/summary` so the view is bookmarkable; the chips just set those query params.
- **Repo identity from PR URL.** Parse `github.com/<owner>/<repo>/pull/N` → `<owner>/<repo>`. Tickets without a merged PR don't contribute to the shipped buckets.
- **Constants documented in code, not env.** Dev-hours-saved heuristic (4h/ticket) is a `const DEV_HOURS_PER_TICKET = 4` with a comment pointing at DEN-2320. Min-runs-per-repo for "worst CI" is `3`.

## Tasks (low-risk first)

### T1 — Backend: getPeriodSummary
- New types in `repository.ts`: `PeriodSummary`, `ThroughputBlock`, `HealthBlock`, `CostBlock`, `ModelCostRow`, `TimeBlock`, `FailureBlock`, `ShippedBlock`, `CheckFailureRow`, `RepoPassRow`, `TicketRef`, `ShippedRepoBucket`.
- `Repository.getPeriodSummary({ from, to }): Promise<PeriodSummary>` added to the interface.
- Implementation does one query per dimension (Drizzle SQL); aggregation in JS so the same body works for both dialects (sqlite sync + pg async via the existing `any`-internals pattern).
- Each block carries a `prior` sub-block computed over the immediately-prior same-length window in one second pass — share the SQL via a private `computeBlocks(handle, from, to)` helper called twice.

### T2 — Backend: route + tests
- `GET /api/summary?from&to` in `src/backend/routes/index.ts`. Defaults: `to = now`, `from = to - 7d`. Validate ISO and `from < to`; 400 on parse failure.
- Tests in `src/backend/routes/api.test.ts` against the seeded mock DB: default window, custom window, throughput math, byModel sort, shipped grouping, prior-period symmetry.

### T3 — UI: api wiring
- `src/ui/src/api/types.ts` — mirror the backend's `PeriodSummary` shape (timestamps as ISO strings, matching the rest of the API).
- `src/ui/src/api/client.ts` — `fetchSummary({ from, to })`.
- `src/ui/src/api/queries.ts` — `useSummary({ from, to })`.

### T4 — UI: page scaffold + period picker
- `src/ui/src/pages/SummaryPage.tsx` with the `PageHeader` + chips + `QueryBoundary`.
- Period state lives in `useSearchParams`; chips set `?from&to`. Custom-range view shows two `<input type="date">`.
- `RefreshButton` on the header.

### T5 — UI: four headline cards
- `src/ui/src/pages/summary/*` — one tiny component file per card (ThroughputCard, HealthCard, CostCard, TimeCard) plus a shared `DeltaChip`.
- 2-up grid on `md+` screens.

### T6 — UI: failures + shipped cards
- `FailuresCard` (full-width): top CI check names list + tickets-at-max-attempts list + worst repos list.
- `ShippedCard` (full-width): count strip + per-repo collapsible groups of ticket links.

### T7 — Nav + route + smoke
- `App.tsx` adds `<NavLink to="/summary">Summary</NavLink>` after the Tickets/Workers/Models trio.
- `<Route path="/summary" element={<SummaryPage />} />`.
- Manual smoke against a seeded SQLite (the mock data has enough rows to exercise all six cards).
