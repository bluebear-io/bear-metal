# DEN-2288: Observability Integration — manager/worker → dashboard DB

## Overview

The observability dashboard (DEN-2271) is currently driven by **mock data** seeded into
SQLite; the backend opens that file **read-only** and the manager/worker write nothing.
This work builds the missing writer: the live manager + worker emit their real ticket /
run / PR / CI / worker state and history into the dashboard DB via **HTTP write endpoints
on the backend**, so the UI shows the real agent instead of a fixture.

This is a **phase-1, persist-what-exists** cut. Data that requires new mechanisms
(attempt caps, worker heartbeats, CI conclusion polling, GitHub PR timestamps) is
explicitly deferred and listed in §8.

**Linear:** [DEN-2288](https://linear.app/bluebearsecurity/issue/DEN-2288/observability-integration) (parent: [DEN-2266](https://linear.app/bluebearsecurity/issue/DEN-2266))

## Context

### Current architecture (origin/main)
- **Manager and worker run in the same process** (`src/manager/index.ts` starts both a
  `Scheduler` and a `TaskWorker`). They coordinate through a **task-queue DB**
  (`bear-metal-manager.sqlite` or Postgres, via `DATABASE_URL`) holding one `tasks` table.
  A task row is effectively a **run**.
- The **dashboard DB** (`src/backend/db`, path `BEAR_METAL_DB_PATH`) is a separate SQLite
  file with 6 tables (`tickets`, `workers`, `runs`, `pull_requests`, `ci_runs`, `events`).
  Today only `src/backend/mock/seed.ts` writes it; `openReadOnlyDb` enforces read-only.
  The `New*` insert types in `db/types.ts` exist but are unused — this is the writer that
  was designed but never built.
- **ID mismatch to reconcile:** the task queue keys tickets by Linear **identifier**
  (`DEN-3001`); `tickets.id` is the Linear **issue id**. The manager holds both on the
  `Ticket` in `TicketContext` at enqueue time.

### Data availability (gap analysis)
Legend: ✅ available/derivable today · ⚠️ partial · ❌ not tracked.

| Table | Status | Notes |
|---|---|---|
| `tickets` | ✅ mostly | `id, identifier, title, description, url, branchName, linearStatus*, labels` come straight from the Linear `Ticket`. |
| `tickets.bmStatus` | ⚠️ derive | No unified enum exists; mapped from manager flow (§4). `ci_running`/`abandoned` deferred. |
| `tickets.attemptCount/maxAttempts` | ❌ | No attempt counting and **no cap** today (scheduler re-dispatches indefinitely). Phase 1: count derived from runs; `maxAttempts` = display-only config constant. |
| `runs` | ⚠️ | Task row maps here. `status` only `pending`/`done` today; `trigger`, `attemptNumber`, `stopReason`, `error` not tracked (worker crashes are only logged). |
| `workers` | ⚠️ | `id` is an ephemeral random UUID per process; `name`, `lastHeartbeatAt` absent; `status` partly derivable (busy/idle). |
| `pull_requests` | ✅ mostly | From GitHub `PullRequest`; the lean type **drops** `createdAt`/`updatedAt`. |
| `ci_runs` | ⚠️ | Only a `testsFailed` boolean + *failed* checks are fetched — no `passed`/`running` conclusion, url, summary, or timestamps. |
| `events` | ❌→✅ | None emitted, but all 13 types map 1:1 to existing log sites in scheduler/dispatch/pi. |

## Design

### 1. Write boundary (backend = sole writer)
- Add `openReadWriteDb(path)` alongside the existing `openReadOnlyDb` in
  `src/backend/db/client.ts`. The write-server process opens RW; the UI continues to read
  the same file. The read router and its read-only client are untouched.
- New write router `src/backend/routes/ingest.ts`, mounted under `/api`, behind a
  **shared-secret bearer guard** (`INGEST_TOKEN`) — `authStub` is a no-op and these routes
  mutate (security-by-design, AGENTS.md). Missing/invalid token → 401.
- Endpoints are **idempotent upserts** keyed on a client-supplied id (drizzle
  `onConflictDoUpdate`), so manager retries are safe:
  - `PUT /api/tickets/:id`
  - `PUT /api/workers/:id`
  - `PUT /api/runs/:id`
  - `PUT /api/pull-requests/:id`
  - `PUT /api/ci-runs/:id`
  - `POST /api/events` (append-only; server-generated id)
- Upsert logic lives in `src/backend/db/writer.ts` (one function per table), validated at
  the route boundary against the table enums. Invalid enum/missing field → 400 (fail fast,
  no defaults).

### 2. Dashboard client (manager/worker side)
- One thin typed HTTP client in `src/shared/dashboard/client.ts` exposing
  `upsertTicket`, `upsertWorker`, `upsertRun`, `upsertPullRequest`, `upsertCiRun`,
  `recordEvent`. Config: `DASHBOARD_URL` + `INGEST_TOKEN`.
- **Non-fatal by design:** a dashboard write failure must never break the agent loop. The
  client logs the error and returns; observability is best-effort. (This is the one
  sanctioned deviation from fail-fast — the dashboard is a read model, not the system of
  record. Stated explicitly here so it is not mistaken for a swallowed bug.)

### 3. Runs — enrich tasks, project to dashboard
- Extend `DispatchTaskInput` / the `tasks` row with `trigger`
  (`new | ci_failure | delegated_back`) and `attemptNumber`. The scheduler **already
  computes** the reason in `decideForOpenPr` / admission (`resuming`, `testsFailed`,
  `hasUnresolvedComments`) — thread it into `enqueue` instead of discarding it.
- `attemptNumber` = count of prior tasks for that ticket + 1 (queried at enqueue).
- Project the run lifecycle at the existing hook points:

  | Hook point | Run write |
  |---|---|
  | `tasks.enqueue` | `status=dispatched`, `trigger`, `attemptNumber`, `createdAt` |
  | `tasks.acquireNext` | `status=running`, `workerId`, `startedAt` |
  | `tasks.complete` (done **or** pending/handback) | `status=succeeded`, `endedAt`, `stopReason=completed` |
  | `TaskWorker` `.catch` | `status=crashed`, `error`, `stopReason=crash`, `endedAt` |

- `failed`, `timed_out`, and `stopReason=timeout` stay unused in phase 1 (no timeout
  mechanism — deferred, §8).

### 4. `bm_status` derivation
Central mapping from the existing flow.

**Reliably emitted in phase 1:**
| `bmStatus` | When |
|---|---|
| `discovered` | Ticket tracked in the store but not yet dispatched |
| `dispatched` | Task enqueued, not yet acquired |
| `in_progress` | Run acquired/running (pi session active) |
| `pr_open` | Worker returned a PR (`DispatchResult.pr` set) and it is open |
| `ci_failed` | Scheduler observed `testsFailed` on the PR |
| `completed` | PR merged → manager `handBack` |

**Deferred (depend on missing mechanisms):** `ci_running` (needs CI conclusion polling) and
`abandoned` (needs the attempt cap). See §8.

### 5. Events
All 13 `events.type` values map 1:1 to existing log sites; each site starts emitting a
structured event (via `recordEvent`) alongside its log line. `source` is derived from the
emit site (`manager` | `worker` | `ci`).

| Event type | Emit site | Phase |
|---|---|---|
| `ticket_discovered` | `admitNewTickets` "picked up ticket" | 1 |
| `dispatched` | `ticket-handler.handle` "enqueueing ticket task" | 1 |
| `branch_created` | worker PR creation path (`pi.ts`) | 1 |
| `progress` | dispatch "pi worker session" milestones | 1 |
| `pr_opened` | worker returns `DispatchResult.pr` | 1 |
| `ci_failed` | `decideForOpenPr` `testsFailed` | 1 |
| `delegated_back` | `decideForOpenPr` resuming / unresolved comments | 1 |
| `ci_started`, `ci_passed` | — | 2 (CI polling) |
| `worker_crashed` | `TaskWorker` `.catch` | 1 |
| `worker_timeout` | — | 2 (no timeout mechanism) |
| `ticket_completed` | refresh "handed ticket back after merge" | 1 |
| `ticket_abandoned` | — | 2 (attempt cap) |

### 6. Workers (phase-1 minimal)
- Write a worker row on `TaskWorker` start (`id` = worker UUID, `name` = `hostname/pid`,
  `status=idle`, `startedAt`). Update `status` `busy`/`idle` on acquire/idle and bump
  `updatedAt` on every write (last-write time is the phase-1 liveness signal).
- `lastHeartbeatAt`, `dead`/`stopped` detection, and a real heartbeat are deferred (§8).

### 7. IDs
| Table | id |
|---|---|
| `tickets` | Linear issue id (manager has it on `TicketContext` at enqueue) |
| `runs` | task UUID |
| `workers` | worker UUID |
| `pull_requests` | `owner/repo#number` |
| `ci_runs` | generated UUID |
| `events` | server-generated UUID |

### 8. Explicitly deferred to phase 2 (the "can't get today" list)
1. **Attempt cap** enforcement + `abandoned` status/event + real `maxAttempts`
   (phase 1 writes a display-only config constant; cap is the documented end-state in
   DEN-2271).
2. **Worker heartbeat** + `lastHeartbeatAt` + `dead`/`stopped`/timeout detection
   (phase 1: `busy`/`idle` only, liveness = last write time).
3. **CI conclusion polling** → `ci_running`/`ci_passed` + ci-run `url`/`summary`/timestamps
   (phase 1: only `failed` recorded from the existing `testsFailed` signal).
4. **GitHub PR timestamps** (phase 1: observation timestamps, since the lean `PullRequest`
   type drops `createdAt`/`updatedAt`).

  rows/events land (the repo requires an integration test where applicable).
- **Unit:** the `bm_status` mapper and the run-status projector (stable behavior, not
  implementation detail).
- Existing read-router and seed tests must stay green.
